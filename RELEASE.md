# Release Process

This document describes how releases are cut for repositories that adopt
the shared tooling in this `.github` repository. Individual projects may
extend or override this process; where they differ, the project's own
documentation takes precedence.

## Versioning

Projects follow [Semantic Versioning](https://semver.org/):

- **MAJOR** — incompatible or breaking changes
- **MINOR** — backward-compatible functionality
- **PATCH** — backward-compatible bug fixes

## Release Drafter

Repositories using the shared `release-drafter.yml` workflow automatically
maintain a draft release that aggregates merged pull requests, grouped by
label, since the last published release. The version bump is inferred from
pull request labels:

| Label            | Resolves to |
| ----------------- | ----------- |
| `breaking-change` | major       |
| `enhancement`      | minor       |
| `bug`, `documentation`, `dependencies` | patch |

## Cutting a release

1. Review the draft release generated by Release Drafter and confirm the
   proposed version and changelog entries are accurate.
2. Edit the draft if any entries are miscategorized or missing.
3. Publish the release from the GitHub UI, which creates the corresponding
   Git tag.
4. If the project publishes packages (PyPI, npm, container images, etc.),
   confirm the project's publish workflow completed successfully after the
   release was published.

## Pre-releases

Pre-release versions (`-alpha`, `-beta`, `-rc.N`) may be published for
significant changes that warrant wider testing before a stable release.
Pre-releases are marked as such in GitHub so they don't appear as the
latest release by default.

## Security releases

Releases that include a security fix follow the coordinated disclosure
process described in [SECURITY.md](SECURITY.md) rather than the standard
process above.
